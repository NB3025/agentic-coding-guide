# 03. Phase별 통합 적용 가이드

> 12개 패턴을 실제 프로젝트에 적용할 때, Phase 순서로 어떻게 작업하는지 상세 기술.
> 각 Phase에서 어떤 패턴이 작동하고, 왜 이 순서인지, 주의사항은 뭔지를 포함.

---

## Phase 0: 프로젝트 셋업 (1회)

### 관련 패턴
- #5 규칙 파일은 짧게, 상세는 분리
- #11 구체적 기술 스택·명령어 명시
- #4 "절대 하지 말 것" 명시
- #12 보안·성능 가드레일

### 왜 이 4개가 묶이는가

이 4개는 전부 "규칙 파일에 뭘 어떻게 넣을까"에 대한 것이다:
- #5가 구조 (얼마나 짧게, 어떻게 분리)
- #11이 내용의 종류 (스택, 명령어)
- #4가 제약 (금지 사항)
- #12가 빠뜨리기 쉬운 것 (보안·성능)

실제로는 **하나의 파일 세트를 만드는 하나의 작업**이다.

### 작업 순서

#### Step 1: 루트 규칙 파일 생성

파일명: `CLAUDE.md` (Claude Code) 또는 `AGENTS.md` (범용)

#5에 따라 60~100줄 이내로 구성:

```markdown
# [프로젝트명]

## Tech Stack (#11)
- [구체적 기술 스택 + 버전]
- [패키지 관리 도구]
- [테스트 명령 — 플래그까지]
- [린트/포맷 명령]

## 세션 시작 규칙 (#1 — 여기서 미리 넣어둠)
- git log --oneline -15 로 최근 작업 확인
- git status 로 현재 상태 확인

## 커밋 규칙 (#2, #8 — 여기서 미리 넣어둠)
- 하나의 논리적 변경 = 하나의 커밋
- 형식: <type>(<scope>): <description>
- 변경 전 반드시 현재 상태 커밋

## Boundaries (#4)
- ✅ Always: [항상 해야 할 것]
- ⚠️ Ask first: [확인 후 진행할 것]
- 🚫 Never: [절대 금지 — 구체적으로]

## Security (#12)
- [이 프로젝트에 해당하는 보안 규칙만]

## Performance (#12)
- [이 프로젝트에 해당하는 성능 규칙만]

## 상세 가이드 (#5의 분리 원칙)
- 완료 전 체크리스트: [스킬 파일 경로]
- 코드 컨벤션: [가이드 파일 경로]
```

#### Step 2: 스킬/가이드 파일 생성

#5의 3-Tier 구조에 따라 상세 내용을 분리:

```
.claude/skills/ (또는 .cursor/rules/ 또는 docs/agent-guides/)
├── self-review/SKILL.md      ← #6 + #9 + #12 통합
└── conventions/SKILL.md      ← 상세 코드 컨벤션
```

#### Step 3: 금지 사항은 최소한만

#3(obviousworks)의 핵심 — 프로젝트 시작 시에는 "아직 뭘 틀릴지 모른다."
그래서 #4의 금지 사항은 **보편적인 것만** 넣는다:

```
🚫 Never:
- 시크릿을 코드에 포함
- 프로덕션 DB에 직접 접근
- 테스트 없이 커밋
```

프로젝트 특유의 금지 사항은 Phase 5에서 실수를 관찰한 후 축적된다.

### 주의사항

Agent READMEs 논문(arxiv 2602.11988)에서 발견:
"불필요한 규칙이 오히려 에이전트 성능을 저하시킨다."

→ **이 프로젝트에 실제로 해당하는 것만** 넣는다.
→ "혹시 모르니까"로 모든 보안 규칙을 복붙하면 안 된다.
→ #12(보안·성능)와 #5(짧게 유지)의 긴장은 **원칙만 루트, 상세는 스킬**로 해결.

## Phase 1: 매 세션 시작

### 관련 패턴
- #1 git log로 컨텍스트 주입

### 왜 단독 Phase인가

#1은 **매 세션마다** 실행되는 유일한 진입점이다.
에이전트가 이전 작업의 맥락을 잡는 핵심 순간이기 때문에 별도 Phase로 분리했다.

### 작업 방법

Phase 0에서 이미 규칙 파일에 넣어뒀으므로, 에이전트가 세션 시작 시 자동 실행:

```
1. git log --oneline -15
2. git status
3. (이전 작업 이어가는 경우) 관련 파일 변경 이력 확인
```

### #8과의 연결

이 시점에서 #8(Conventional Commits)의 효과가 드러난다.
커밋 메시지가 구조화되어 있으면 git log만으로 충분.
메시지가 vague하면 diff까지 봐야 해서 컨텍스트 낭비.

### #10과의 연결

히스토리가 정리되어 있으면(#10) 노이즈 없이 핵심만 시딩.
정리 안 되어 있으면 `fix`, `fix again`, `revert` 같은 커밋이 혼란을 줌.

## Phase 2: 작업 중

### 관련 패턴
- #2 Git Checkpointing
- #8 Conventional Commits

### 왜 이 2개가 묶이는가

#2는 "자주 커밋하라", #8은 "커밋 메시지를 구조화하라".
이 둘은 동시에 작동한다 — 커밋을 할 때마다 적용되니까.

### 작업 방법

규칙 파일에 이미 포함되어 있으므로 에이전트가 자동 준수:

```
작업 흐름:
1. 현재 상태 커밋 (checkpoint)
2. 변경 작업 수행
3. 논리적 단위 완성 → 커밋 (Conventional Commits 형식)
4. 반복

커밋 형식:
feat(auth): add JWT refresh token rotation
fix(api): handle null response from payment gateway
test(auth): add edge cases for expired tokens
refactor(db): extract query builder to separate module
docs(api): update endpoint documentation for v2
chore(deps): update FastAPI to 0.115.3
```

### #7과의 연결 (미래 의존)

지금 잘게 커밋하는 것이 나중에 문제 발생 시 #7(Bisect)의 정밀도를 결정한다.
커밋 하나가 10개 파일을 바꾸면 bisect로 원인 커밋을 찾아도 어느 변경이 문제인지 모른다.
커밋 하나가 하나의 논리적 변경이면 bisect 결과가 즉시 원인을 가리킨다.

## Phase 3: 작업 완료 시

### 관련 패턴
- #6 Self-Review Checklist
- #9 코드 변경 시 문서 자동 업데이트
- #12 보안·성능 (재검증)

### 왜 이 3개가 묶이는가

전부 "커밋/PR 전에 반드시 거치는 관문"이다.
#6이 코드 품질 체크, #9가 문서 동기화, #12가 보안·성능 재검증.
#9는 #6의 체크리스트 안에 하나의 항목으로 포함 가능.

### 작업 방법

스킬 파일(self-review/SKILL.md) 또는 규칙 파일에서 참조:

```
## 작업 완료 전 체크리스트

### 기능
- [ ] 변경한 코드가 의도대로 동작하는가?
- [ ] 엣지 케이스를 처리했는가?

### 테스트
- [ ] 새 코드에 대한 테스트를 작성했는가?
- [ ] 기존 테스트가 모두 통과하는가? (명령: [Phase 0에서 정의한 테스트 명령])

### 보안 (#12 재검증)
- [ ] 사용자 입력을 검증하는가?
- [ ] 시크릿이 코드에 포함되지 않았는가?

### 성능 (#12 재검증)
- [ ] N+1 쿼리가 없는가?
- [ ] 불필요한 데이터를 로딩하지 않는가?

### 문서 (#9)
- [ ] 관련 문서를 업데이트했는가?
- [ ] API가 바뀌었으면 API 문서도 바뀌었는가?

### 규칙 준수
- [ ] 금지 사항(🚫)을 위반하지 않았는가?
```

### 구성 의도

#12(보안·성능)가 여기서 다시 나타나는 이유:
Phase 0에서 규칙으로 정의해도 에이전트가 작업 중에 놓칠 수 있다.
Phase 3에서 체크리스트로 **한 번 더 확인**하면 놓친 것을 잡아낸다.
이것이 #12가 "관통 패턴"인 이유 — 정의(Phase 0)와 검증(Phase 3) 두 지점에서 작동.

## Phase 4: 문제 발생 시

### 관련 패턴
- #7 Git Bisect로 회귀 탐색

### 왜 별도 Phase인가

#7은 다른 패턴과 달리 "매번" 실행하는 게 아니라 "문제가 생겼을 때만" 쓴다.
하지만 그 효과는 #2(Checkpointing)와 #8(Conventional Commits)에 강하게 의존한다.

### 작업 방법

규칙 파일에 넣기보다 문제 상황에서 에이전트에게 직접 지시:

```
"테스트 X가 실패하고 있어. 언제부터 실패했는지 git bisect로 찾아줘."
```

에이전트가 자동으로:
1. 테스트 조건 설정
2. 시작/끝 커밋 범위 지정
3. 이분 탐색 실행
4. 원인 커밋 식별

### 전제 조건

- Phase 2의 #2에 의해 커밋이 충분히 잘게 쌓여 있어야 함
- Phase 2의 #8에 의해 커밋 메시지가 구조화되어 있어야 원인 파악이 쉬움

→ #7은 Phase 2에서 쌓은 자산(잘게 커밋 + 좋은 메시지)을 소비하는 패턴.

## Phase 5: 사후 개선

### 관련 패턴
- #3 반복 실수 관찰 → 규칙 추가
- #10 Git 히스토리 편집

### 왜 이 2개가 묶이는가

둘 다 "피드백 루프를 닫는" 패턴이다:
- #3은 규칙 파일의 피드백 루프 (실수 → 규칙 추가 → Phase 0으로)
- #10은 git 히스토리의 피드백 루프 (정리 → Phase 1의 시딩 품질 향상)

### 작업 방법 — #3 (실수→규칙)

```
에이전트가 실수함
    ↓
실수 패턴 식별 ("또 None 반환", "또 N+1 쿼리")
    ↓
규칙 파일의 🚫 Never 섹션에 추가 (#4 형식)
    ↓
다음 세션에서 에이전트가 규칙 읽음 (#1)
    ↓
같은 실수 방지
```

월 1회 규칙 리뷰 (obviousworks 원안):

```
- 이 규칙이 아직 유효한가? → 기술 변경으로 해당 없어진 건 삭제
- 단순화할 수 있는가? → 3개 규칙이 하나로 합쳐질 수 있는지
- 중복이 있는가?
- 새로 추가할 게 있는가?
- 파일 크기가 #5의 한도를 넘지 않는가?
```

### 작업 방법 — #10 (히스토리 편집)

PR 머지 전 또는 마일스톤 후:

```
- 실험적 커밋들을 squash
- 잘못된 커밋 메시지 수정
- 의미 없는 연쇄 커밋 합치기

에이전트에게: "이 브랜치의 커밋들을 논리적 단위로 정리해줘"
```

### Phase 5 → Phase 0 루프

```
Phase 5                          Phase 0
┌──────────────────┐            ┌─────────────────┐
│ #3 실수 관찰      │───축적───→│ #4 금지 사항 추가  │
│ #10 히스토리 정리  │───정리───→│ #1 시딩 품질 향상  │
│ (월 1회 규칙 리뷰) │───삭제───→│ #5 규칙 간결화    │
└──────────────────┘            └─────────────────┘
```

이 루프가 시간이 지날수록 에이전트가 프로젝트에 맞춰지는 메커니즘이다.
에이전트 자체가 학습하는 게 아니라, **규칙 파일이 진화**하면서
매 세션 더 정확한 컨텍스트를 주입하게 된다.
