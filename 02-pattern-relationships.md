# 02. 패턴 간 연관성 분석 + 구성 의도

> 12개 패턴은 독립적으로 존재하지 않는다. 서로 의존하고, 강화하고, 전제 조건이 된다.
> 이 문서는 그 연결 관계를 분석하고, 왜 이런 순서와 구조로 구성했는지 설명한다.

---

## 1. 생명주기 위에 매핑

12개 패턴을 프로젝트 생명주기에 놓으면 자연스럽게 5개 Phase로 나뉜다:

```
[Phase 0: 프로젝트 셋업] ← 1회
  #5  규칙 파일은 짧게, 상세는 분리
  #11 구체적 기술 스택·명령어 명시
  #4  "절대 하지 말 것" 명시
  #12 보안·성능 가드레일

[Phase 1: 매 세션 시작] ← 매번
  #1  git log로 컨텍스트 주입

[Phase 2: 작업 중] ← 매번
  #2  Git Checkpointing
  #8  Conventional Commits

[Phase 3: 작업 완료 시] ← 매번
  #6  Self-Review Checklist
  #9  코드 변경 시 문서 자동 업데이트

[Phase 4: 문제 발생 시] ← 필요 시
  #7  Git Bisect로 회귀 탐색

[Phase 5: 사후 개선] ← 주기적
  #3  반복 실수 관찰 → 규칙 추가
  #10 Git 히스토리 편집
```

### 구성 의도

이 순서는 내가 만든 게 아니라, 각 패턴의 소스에서 전제 조건을 따져보면
자연스럽게 이 순서가 나온다. 예를 들어:

- #7(Bisect)은 #2(Checkpointing)가 없으면 동작하지 않음 → #2가 먼저
- #1(git log 시딩)은 #10(히스토리 정리)이 되어 있어야 효과적 → #10이 #1에 선행
- #3(실수→규칙)은 #4(금지 사항)의 내용을 채우는 프로세스 → #4가 형식, #3이 내용

각 패턴의 출처는 독립적이지만(Simon Willison, Groff.dev, GitHub Blog 등),
이들이 설명하는 것은 같은 개발 사이클의 서로 다른 지점이다.

---

## 2. 의존 관계 상세

### #2(Checkpointing) → #7(Bisect)

**관계**: #2가 #7의 전제 조건

Simon Willison은 bisect를 설명하면서 "커밋이 쌓여 있어야 이분 탐색이 가능하다"는 점을
암묵적으로 전제한다. 커밋이 듬성듬성하면 bisect가 넓은 범위에서만 원인을 좁힐 수 있다.
커밋이 잘게 쌓여 있으면 정확한 원인 커밋까지 찾을 수 있다.

→ **#2의 정밀도가 #7의 정밀도를 결정한다.**

### #8(Conventional Commits) → #1(git log 시딩)

**관계**: #8이 #1의 품질을 결정

Simon Willison이 #1에서 말한 것 — git log로 세션을 시딩하면 "코드 변경분 + 커밋 메시지"가
컨텍스트에 들어간다. 여기서 커밋 메시지가 `feat(auth): add JWT validation`이면
즉시 파악되지만, `update code`면 에이전트가 diff까지 봐야 해서 컨텍스트가 낭비된다.

→ **#8이 좋으면 #1에서 git log만으로 충분. #8이 나쁘면 #1에서 diff까지 봐야 함.**

### #10(히스토리 편집) → #1(git log 시딩)

**관계**: #10이 #1의 노이즈를 제거

Simon Willison은 #10에서 히스토리를 "의도적으로 편집된 이야기"라고 했다.
히스토리가 `fix`, `fix again`, `actually fix`, `revert fix`로 채워져 있으면
다음 세션에서 git log로 시딩할 때 에이전트가 혼란스러워한다.

→ **#10으로 정리해두면 #1의 시딩 품질이 올라간다.**

### #4(금지 사항) ↔ #3(실수→규칙)

**관계**: 피드백 루프 (형식 ↔ 프로세스)

#4(GitHub Blog, Groff.dev)는 "금지 사항을 적어라"라는 **형식**을 제공한다.
#3(obviousworks)는 "금지 사항을 어떻게 발견하나"라는 **프로세스**를 제공한다.

프로젝트 시작 시에는 #4의 금지 사항이 거의 비어 있다.
에이전트를 운용하면서 #3의 관찰 프로세스를 통해 실수를 발견하고,
그걸 #4의 형식에 맞춰 규칙 파일에 축적한다.

→ **#4가 그릇이고 #3이 내용물. 둘이 합쳐져야 실수 방지가 실제로 작동한다.**

### #12(보안·성능) → #6(Self-Review) + Phase 0

**관계**: #12는 관통 패턴 — 두 지점에서 나타남

Agent READMEs 논문의 #12는 "보안·성능 규칙을 넣어라"인데,
이것은 Phase 0(규칙 정의)에서 한 번, Phase 3의 #6(체크리스트)에서 한 번,
총 두 번 적용된다.

- Phase 0: 규칙 파일에 보안·성능 섹션을 넣음 (에이전트가 처음부터 인지)
- Phase 3: 체크리스트에서 "보안 규칙 위반 없는지?" 재검증 (놓친 것 잡아냄)

→ **#12는 독립 패턴이라기보다 Phase 0과 Phase 3에 걸치는 관통 요소.**

### #6(Self-Review) ↔ #9(문서 업데이트)

**관계**: 같은 시점, 합칠 수도 있음

#6(Groff.dev, obviousworks)과 #9(Blake Niemyjski)는 둘 다
"작업 완료 전에 해야 할 것"이다. 분리해도 되고, #6의 체크리스트 안에
"관련 문서를 업데이트했는가?"를 포함시키면 #9가 #6의 한 항목이 된다.

→ **어느 쪽이든 커밋/PR 전에 둘 다 통과해야 한다는 게 핵심.**

## 3. Phase 5 → Phase 0 피드백 루프

가장 중요한 연결은 **Phase 5에서 Phase 0으로 돌아가는 루프**다.

```
Phase 5                          Phase 0
┌──────────────────┐            ┌─────────────────┐
│ #3 실수 관찰      │───축적───→│ #4 금지 사항 추가  │
│ #10 히스토리 정리  │───정리───→│ #1 시딩 품질 향상  │
│ (월 1회 규칙 리뷰) │───삭제───→│ #5 규칙 간결화    │
└──────────────────┘            └─────────────────┘
```

이 루프가 "에이전트가 시간이 지날수록 좋아지는" 메커니즘이다:

1. 실수 발생 → 규칙 추가 (#3 → #4) → 같은 실수 방지
2. 규칙이 쌓임 → 불필요한 것 삭제 (#3의 월간 리뷰 → #5의 간결화)
3. 히스토리 정리 (#10) → 다음 세션 시딩 품질 향상 (#1)

### 구성 의도

이 피드백 루프 구조는 내가 만든 것이 아니다.
- #3의 출처(obviousworks)가 "관찰 → 규칙 추가 → 월간 리뷰" 사이클을 명시
- #10의 출처(Simon Willison)가 "히스토리를 의도적으로 편집"하라고 명시
- #1의 출처(Simon Willison)가 "git log로 세션 시작"을 명시

이 세 가지를 시간 축 위에 놓으면 자연스럽게 루프가 보인다.
내가 한 것은 **이 루프를 명시적으로 그린 것**뿐이다.

## 4. 연결 강도 요약

| 패턴 A | 패턴 B | 관계 | 강도 |
|--------|--------|------|------|
| #2 Checkpointing | #7 Bisect | 전제 조건 | **필수** — #2 없이 #7 불가 |
| #8 Conv. Commits | #1 git log 시딩 | 품질 결정 | **강함** — #8 없으면 #1 비효율 |
| #10 히스토리 편집 | #1 git log 시딩 | 노이즈 제거 | **강함** — #10 없으면 #1에 노이즈 |
| #3 실수→규칙 | #4 금지 사항 | 피드백 루프 | **필수** — #4의 내용을 #3이 채움 |
| #12 보안·성능 | #6 Self-Review | 관통 패턴 | **강함** — #12가 #6의 검증 항목 |
| #6 Self-Review | #9 문서 업데이트 | 같은 시점 | **선택적** — 합치거나 분리 가능 |
| #5 짧은 규칙 | #12 보안·성능 | 상충 관리 | **주의** — #12 넣되 #5 한도 내에서 |

### #5와 #12의 긴장

유일하게 서로 **상충**하는 관계가 있다:
- #5는 "규칙 파일을 짧게 유지하라" (60~100줄)
- #12는 "보안·성능 규칙을 넣어라" (빠뜨리면 위험)

둘 다 정당한 근거가 있다. 해결 방법은 #5의 3-Tier 구조:
보안·성능의 **원칙만** 루트 파일에 넣고 (2~3줄),
**상세 규칙**은 스킬 파일로 분리한다.

## 5. 이 구조에서 CHANGELOG.md의 위치

12개 패턴 중 "CHANGELOG.md"를 직접 다루는 패턴은 없다.
Keep a Changelog (keepachangelog.com)라는 업계 표준이 별도로 존재하고,
#9(Blake Niemyjski)의 releasenotes 스킬이 가장 가까운 소스다.

CHANGELOG.md가 이 체계에서 가장 자연스럽게 들어가는 위치는:

1. **Phase 3의 #9 확장** — 코드 변경 시 CHANGELOG도 업데이트
2. **Phase 5의 #3 보조** — 실수·롤백을 CHANGELOG에 기록하면 #3의 관찰이 쉬워짐
3. **Phase 1의 #1 보조** — git log + CHANGELOG를 함께 읽으면 "왜"까지 파악 가능

단, CHANGELOG 자체가 검증된 패턴인지, 내 조합인지를 구분해야 한다:
- **검증된 것**: Keep a Changelog 표준 (업계 표준), releasenotes 스킬 (Blake 실제 운용)
- **내 조합**: CHANGELOG에 🚫 마커를 붙이고 교훈 필드를 추가하는 것, 실수 기록 3요소

이 점을 프로젝트 적용 시 인지하고 사용해야 한다.
