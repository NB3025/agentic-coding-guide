# Pattern 5: Short Rules

## 문제

규칙이 길면 에이전트가 읽지 않는다. 컨텍스트 윈도우를 낭비하고, 핵심이 묻힌다.

## 해결

규칙은 **짧고 구체적**으로 작성한다.

## 나쁜 예

```markdown
코드를 작성할 때는 가능한 한 기존의 코드 스타일과 일관성을 유지하도록 노력해야 하며,
특히 변수 네이밍이나 함수 구조 등에서 프로젝트의 기존 패턴을 존중하는 것이 중요합니다.
또한 새로운 파일을 추가할 때는 기존 디렉토리 구조를 따라야 합니다.
```

## 좋은 예

```markdown
- 변수: camelCase, 함수: camelCase, 클래스: PascalCase
- 새 파일은 기존 디렉토리 구조를 따름
```

## 기준

- 한 규칙 = 1~2줄
- "왜"보다 "뭘 하라"에 집중
- 애매한 표현 금지: "가능한 한", "적절하게", "필요 시"
- 구체적 예시가 추상적 설명보다 낫다
