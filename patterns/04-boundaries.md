# Pattern 4: Boundaries

## 문제

에이전트가 뭘 해도 되고 뭘 하면 안 되는지 모른다. "알아서 잘 해줘"는 사고의 원인.

## 해결

**3단계 경계**를 명시한다.

## 3단계

### ✅ Always
무조건 해야 하는 것. 예외 없음.

```markdown
- 테스트 작성 후 커밋
- 입력 검증 (사용자 입력, API 응답)
- 환경변수로 시크릿 관리
- 에러 핸들링 포함
```

### ⚠️ Ask first
혼자 판단하지 말고 물어볼 것.

```markdown
- 기존 아키텍처 변경
- 새 외부 의존성 추가
- DB 스키마 변경
- 기존 API 인터페이스 변경
```

### 🚫 Never
절대 하지 말 것. 이유 불문.

```markdown
- 시크릿을 코드에 하드코딩
- 프로덕션 DB에 직접 접근
- 테스트 없이 커밋
- 기존 테스트 삭제 (수정은 OK)
```

## 왜 3단계?

- 2단계(해/하지마)는 너무 이분법적 → 에이전트가 회색 영역에서 멈춤
- "Ask first"가 있으면 에이전트가 불확실할 때 물어볼 수 있음
- 프로덕션 사고의 대부분은 "Ask first" 영역에서 혼자 판단했을 때 발생

## 이 파일은 진화한다

프로젝트 초기에는 보편적 규칙만. Pattern 3 (Mistake → Rule)에 의해 실수가 규칙으로 추가된다.
